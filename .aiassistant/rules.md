# Правила разработки для AI Chat (PhpStorm)

## 1) Принципы кода
- Писать только читаемый и поддерживаемый код: короткие методы, понятные имена, минимум скрытой магии.
- Соблюдать SOLID, DRY, KISS, разделение ответственности по слоям.
- Не смешивать бизнес-логику, инфраструктуру и представление в одном классе.
- Любое нетривиальное решение сопровождать кратким комментарием "почему".

## 2) Архитектура и файлы
- Разносить код по логичным модулям: `Controller`, `Service`, `Repository`, `Database`, `View`.
- Каждая сущность и сервис — в отдельном файле.
- Запрещено дублирование regex-логики и SQL: выносить в переиспользуемые компоненты.
- Все настройки хранить централизованно (env + таблица `settings`).

## 3) Парсинг и безопасность
- Парсер реализуется в отдельном Node.js сервисе на Puppeteer + stealth.
- Веб-приложение (PHP) общается с парсером только через четкий контракт (HTTP JSON API).
- Хранить полный путь страницы и нормализованный текст отдельно.
- Маскировка парсера обязательна: случайный user-agent, viewport, тайминги, управление заголовками.

## 4) Фоновые задачи
- Фоновая обработка должна быть идемпотентной: повторный запуск не ломает данные.
- Для сайта всегда фиксировать статус обхода (`idle/running/failed`) и время последнего запуска.
- Параллелизм должен регулироваться настройками (`worker_batch_size`, `crawler_max_pages`, `crawler_depth`).

## 5) Тесты и контроль качества
- Для критичных частей (загрузка regex, матчинг, дедупликация результатов) обязательны unit-тесты.
- Любая новая фича должна иметь минимум один тест на успешный сценарий.
- Перед сдачей: линт/проверка синтаксиса и запуск тестов.

## 6) UI и админка
- Админка должна позволять: добавлять/редактировать сайты, запускать сканирование, смотреть отчеты по вхождениям.
- Отчеты строятся из БД и показывают: сайт, URL страницы, кого нашли, категория, время.

## 7) Ограничения
- Не писать одноразовые "быстрые" решения в обход архитектуры.
- Не добавлять зависимость без практической необходимости.
- Не хранить секреты в коде и репозитории.
